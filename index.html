Hello World

发表于 2022-10-08 更新于 2022-10-26 分类于 随笔感想

从昨天晚上搞到现在，终于勉强把这个写完了。怎么说呢，从最开始那个像是上世纪不知名的古早网站，到现在终于有一个像模像样的个人网站，不得不说还是很有成就感的。学计算机也已经两年了，从最开始连键盘都摸不利索，到现在对于计算机有一个较为粗浅的整体的认知，回头看过去，应当还是有一些成长的吧。算是一个小总结，也算是一个新的开始！

你到这里来做什么

高中给我留下的回忆只有日复一日的联系，为数不多的快乐记忆是和一众男生在教室背后打闹。很幸运，我高考可以算是超常发挥，这对于不太善于应对紧张情绪的我来说，还是很难得的。然后就是填写志愿，糊里糊涂地选择了ZJU，现在想来，也算是误打误撞了吧。但是却没有后悔，不管是浙江大学还是计算机这个专业，都没有让我后悔过。

你要成为什么样的人

如果说在上大学前我没有认真想过这个问题，上了大学之后我发现，就算我认真想这个问题，我依然想不出来。对的我不知道我想成为什么样的人，即使是作为一个已经大三的人，或许未来会知道吧，但是这种事情又何必着急呢？多少人一辈子都没想明白呢！

那些灿若星尘的人

我很感谢那一位位曾给过我力量的人（此处手动@冰冰的小冰，尽管你不认识我，但是很感谢你将你的故事，别人的故事分享，它们就像浩瀚星辰中的一束光，照亮出我的路途，而非那犹如黑洞般漆黑的前路）

我不知道我能否成为那个灿若星辰的人，但是我希望我能成为照亮自己的那盏灯，如果可以，再多照亮我身边的那几个人，这便够了。

之前一直有写博客，或者说写随笔地想法，想在倒是通过这种方式实现了，希望能够坚持写下去。

未来可期

当我一次又一次地思索我的道路究竟在何方的时候，大把的光阴已经悄然流逝，与其被动地等待它来到我的面前，不如抓紧现在的机会，勇敢地去寻找，万一发现了另外一扇窗呢？那就又是意外之喜了。

说回双城之战，里面的每个人同样都有自己独特的性格，说到这里又不得不说一下，这些真正好的动画无一不是在人物的塑造方面各有特色。

另一个印象深刻的是维克多，我本来以为他已经黑化了，在经历不断地身体衰弱和打击之后，但是，在提到制造武器的时候，他是最坚决反对的，他在最后也意识到这条路已经不在正确了，他选择了自己的归宿。

计算机网络-Lecture 1：概论

发表于 2022-10-17 更新于 2022-10-26 分类于 中科大计网

1. 什么是Internet

从具体组成角度

  • 网络：图（节点，边）
  • 计算机网络：由计算机组成节点的网络（路由器，交换机）
    • 节点： 路由器，交换机(end system)
    • 边： 接入网链路，主干链路
    • 协议： 两个节点如何交流
  • 网络的网络

    从服务的角度

  • 通过网络连接起来的分布式应用
  • 通信基础设施为应用提供网络传输服务

2. 网络边缘

网络结构

  • 网络边缘
    • 主机
    • 应用程序
  • 网络核心
    • 路由器
    • 交换机
  • 网络接入
    • 连接边缘与核心的部分

各自的功能

  • 网络应用， 有联网的需求
  • 主机，各种协议栈，为应用联网提供服务
  • 网络核心：传输数据到目的地
  • 网络接入: 将网络连接成更大的网络

网络边缘

  • 端点的主机
  • 边缘交流的模式
    • cs（client-server模式）： 服务器为客户端提供服务
      • 当客户端增大时，性能下降
    • p2p（peer-to-peer模式）：客户端与客户端提供数据，但是仍然有一个服务器进行调j度

网络应用采用的服务(由网络基础设施提供)

采用网络设施的面向连接的服务

  • 进程在通信之前，需要各个层面都知道已经建立连接
  • tcp

    采用基础设施的无连接服务

  • 直接进行发送，网络边缘并不知道对方的状态
  • udp

3. 网络核心

由路由器组成的网状网络

网络的类型

  • 电路交换：专有的线路进行通信，例如电话网
  • 分组交换：将需要传输的数据分成一个个分组，然后交由路由器进行传输

电路交换

在两个路由器之间将线路分为若干条线，独享的线路

  • 特点
    • 不共享
    • 保证了性能
    • 要求建立连接
  • 分割线路的类型（网络资源被分为“片”， 每个片要么空闲，要么被独享）
    • FDM 频分(将频率的一段分为小段)
    • TDM 时分(将一个周期分为小段)
    • 波分(将光波根据波长分为多个段)
  • 传播延迟
  • 电路交换不适合计算机之间的通信
    • 建立时间长
    • 计算机通信由突发性，随时可能发生或者不发生，会浪费“片”
    • 可靠性不高，每一对交换机一旦出现问题，则会直接失败

分组交换

  • 通过网络核心时，每一条使用全部带宽
  • 存储-转发模式，以packet为单位，全部存储后再转发（区分于电路交换，即多个通信可共用）

    存储-转发

  • 必须等到完全接受之后才能进行转发

    排队延迟和丢失

  • 如果前面还没传输完，则需要等待
  • 如果交换机缓存已满，则会被丢弃

    网络核心的关键功能

  • 路由：决定分组采用的源到目标的路径–路由算法
  • 转发：将分组从路由器的输入链路转移到输出链路

    统计多路复用

  • 没有固定的复用方式

分组交换 vs 电路交换

分组交换允许更多用户同时使用网络

  • 适合突发性数据传输
    • 资源共享
    • 无需呼叫
  • 过度使用会导致拥塞
    • 相应的机制

分组交换的类型

  1. 数据报网络
    • 在分组交换机处决定下一跳的目标
    • 问路
    • Internet
  2. 虚电路网络
    • 每个分组带有标签，决定下一跳
    • 呼叫时决定路径并保持不变
    • 路由器维持每个呼叫的状态信息
    • 信令

4. 接入网和物理媒体

层级结构

6. 分组延时，丢失，吞吐量

分组延时

  1. 节点处理延时：分组到达之后需要进行处理
  2. 排队延时：需要等待其他的分组处理完成
  3. 传输延时：L/R 从主机输出bit的时间
  4. 传播延时([[Lec1 概论#排队延迟和丢失|处于网络核心中的分组延时]]): 在线路上传播的时间，如果距离近则可以忽略不记
  • 排队延时
    • R = 链路带宽
    • L = 分组长度
    • a = 分组到达队列的平均速率
    • 流量强度 = $La/R$
      • 当流量强度趋近于1的时候，延时趋近于无限大

分组丢失

  • 缓存队列有限
  • 可能会被上一个节点或源端系统重传，或者不重传

吞吐量

在源端与目标端之间传输的速率

  • 瞬间，平均
  • 短板效应， 在一条链路上，取决于吞吐量小的
  • 实际取决于每一条链路除以同时使用的数量的最小值 即 $min{R_i/N_i}$

7. 协议层次及服务模型

协议与服务

协议层次

  • 将网络的复杂功能分为不同层次的功能，每一层接口为上一层提供接口，无需担心下层的实现
  • 例子，哲学家的交流， 三层（哲学家，翻译，传报人）

服务： 下层向上层提供的功能

  • 服务访问点(SAP): 区分上层的用户（调用者）
  • 通过原语向上层提供服务，可以理解为API
  • 服务的类型
    • 面向连接的服务（需要都知道已经建立了连接，为后续链接做好准备
    • 无连接的服务：不需要建立连接

服务与协议的区别

  • 服务是垂直的关系，下层向上层提供服务
  • 协议：对等层实体进行交流

服务与协议的关系

  • 协议借助下层服务实现
  • 本层实体为上层提供更高级别的服务

数据单元

  • 术语
    • SAP: service access point
    • IDU: interface data unit
    • SDU: service data unit
    • PDU: protocol data unit
    • ICI: interface control information
  • $SDU_n + ICI_n = PDU_n = SDU_{n+1}$

分层实现的好处

  • 结构清晰
  • 便于维护和升级：分别实现
  • 坏处：效率降低

互联网协议栈

  • 应用层：网络应用
    • FTP, HTTP, DNS, SMTP
  • 传输层：主机之间的数据传输
    • 进程到进程的区分，将通信由不可分变为可分
    • TCP，UDP
  • 网络层：end-to-end的数据传输
    • IP
  • 链路层：在相邻两点传输以帧为单位的数据
    • point-to-point: PPP, wifi, ethenet
  • 物理层：将电磁波信号与01数据相互转化

ISO/OSI 参考模型

  • 七层： 应用层，表示层，会话层，传输层，网络层，链路层，物理层
  • 封装与解封装
    • 交换机：链路层，查看网卡物理地址，选择由哪个端口收发
    • 路由器：网络层，查看IP地址，选择由哪个网卡分发

各层次协议数据单元

  • 应用层：报文(message)
  • 传输层：报文段(segment)
  • 网络层：分组(packet),无连接(datagram)
  • 数据链路层：帧(frame)
  • 物理层：比特(bit)

1. 应用层协议原理

1.1 [[Lec1 概论#网络边缘|网络应用的体系结构]]

  • CS client/server
    • 服务器：一直运行
      • 固定的IP地址以及约定好的端口号
      • 扩展性(差)
    • 客户端
      • 主动与服务器通信
      • 动态IP地址
  • Peer to peer
  • 混合模式

1.2 进程通信

  • 进程：在主机上运行的应用程序
    • 客户端进程：发起通信的进程
    • 服务器进程：等待连接的进程
  • 通信
    • 同一个主机内，操作系统定义
    • 不同主机：交换报文
  • 需要解决的问题
    • **进程标识和寻址问题
    • **传输层-应用层提供的服务是什么(SAP socket, API)
    • **如何使用传输层提供的服务，实现应用进程之间的报文交换

1.3 对进程进行编址

  • 进程在发送接受报文的时候需要有唯一标识
    • 主机：IP
    • 采用的协议
    • 端口号：port number
  • HTTP: TCP 80
  • 一个进程，用IP+port表示端节点

1.4 传输层提供的服务

1.4.1 需要穿过层间的信息

  • 必须携带的信息
    • 数据
    • 谁传的(IP + TCP(UDP)端口)
    • 传给谁，对方的(IP + TCP(UDP)端口)

1.4.2层间信息的代表(TCP)

  • API传输报文容易出错
  • 用socket(一个整数)标识通信的双方（类比os句柄fd）
  • TCP socket: 源IP(给下层IP协议使用[[Lec1 概论#数据单元|ICI + SDU = PDU]])，源端口，目标IP，目标端口
  • 本地的标识，其他主机不知道

1.4.3层间信息代码(UDP)

  • UDP两个进程无需建立连接
  • UDP socket： 本IP，本端口
  • 传输报文时需要提供对方IP，port

1.4.4TCP 与 UDP socket间的区别

  • UDP socket 只有自己的IP和socket，相当于唯一标识了一个进程，而TCP socket 相当于在进程基础上表示了一对进程，即一个关系
  • UDP 在调用API时需要向下传递目标IP和目标端口号

1.5 如何使用传输层的服务实现应用

  • 定义应用层协议
  • 编制程序

1.5.1 应用层协议

定义了不同端系统上进程如何相互交换报文
一系列语法和约定好的规则

1.5.2 需要什么样的服务

  • 数据丢失率
  • 吞吐量
  • 延迟
  • 安全性

1.6 传输层提供的服务

  • TCP：可靠，流量控制， 拥塞控制

  • UDP：不可靠

  • UDP存在的必要性，区分不同进程，无需建立连接，不需要可靠性的工作，速度有保证

  • 安全TCP

    • TCP和UDP都没有加密，通过明文传输
    • SSL在TCP上面实现，提供加密服务，在应用层

2. web and HTTP

2.1 一些术语

  • Web页，有一些对象组成，对象可以是html，jpg，java程序
  • Web含有一个base html， 链接其他的对象
  • 通过URL对每个对象进行引用

2.2 HTTP概况

HTTP: hypertext transport protocol

  • 服务器/客户模式
  • HTTP/1.0 HTTP/1.1
  • 使用TCP

    2. 服务器接受连接
    3. 发送HTTP请求或响应
    4. TCP连接关闭
  • **HTTP是无状态的

2.3 HTTP连接

2.3.1 非持久HTTP

  1. HTTP客户发出请求80

2.3.2 持久HTTP

  • 服务器相应之后仍然保持TCP连接
  • 流水方式的持久HTTP
    • 有一个引用就发，不管上一个是否收到
  • 非流水方式的持久HTTP
    • 客户端只有在收到前一个的响应之后才能发出新的请求

2.3.3 响应时间模型

  • 往返时间RTT：一个分组从客户端到服务器再回到客户端的时间
  • 非持久：2个RTT
    • 一个用于建立TCP连接
    • 一个用于请求响应
  • 持久：1+N
    • 一个用于建立TCP连接
    • 每个请求一个RTT

5.1.2 DNS总体思想目标

  • 主要思路

5.2 解析问题-名字服务器

  • 一个命名服务器问题
    • 可靠性，一旦崩溃整个互联网都会崩溃
    • 扩展性，通信容量不能无限扩大
    • 维护，集中式数据库，需要频繁写入更新
  • 区域
    • 域的划分由管理者决定
    • 不同的区域互相不交互
      • 权威服务器，保存其管理的所有域的ip地址

        5.2.1 区域名字服务器维护资源记录

  • 资源记录
    • 作用：维护域名IP的映射关系
    • 储存在数据库中
  • RR格式
    • domain_name：域名
    • TTL：生存时间
    • calss：类别（IN）
    • value：值
    • type：类别资源的类型
  • 四种
    • type = A， name为主机域名，value为ip地址
    • type = CNAME， name为别名，value为规范名字
    • type = NS， name为域名，value为响应的权威服务器的域名
    • type = MX，value为name对应的邮件服务器的名字

5.2.3 DNS大致工作过程

  • 应用解析器向name server发出查询报文
  • name server返回响应报文

5.2.4 本地名字服务器(local name server)

  • 当主机发起DNS查询时，查询发送到本地DNS代理器
  • 先看本地是否有缓存，没有再请求
  • 查询
    • 递归查询，一个一个往下问
    • 迭代查询，一个一个问，每个都会返回部分结果，进而查询
  • 缓存提高效率，删除保持更新

5.2.5 DNS协议、报文

  • flags：查询/回答, 希望递归，递 归可用，应答为权威

    • 增加权威服务器到ip地址的映射

6. P2P应用

6.1 纯P2P架构

  • 没有一直运行的服务器
  • 利用peer的服务能力
  • 应用
    • 文件分发bittorrent
    • 流媒体kankan
    • VoIP
  • 非结构化P2P vs 结构化P2P(DHT)

6.2 文件分发

  • 问题：从一台服务器分发文件(大小为F)到N个peer需要的时间
  • C/S模式
    • 服务器传输, 一个copy： $F/U_s$， N个copy：$NF/u_s$
    • 客户端，每个客户端需要下载， $F/d_{min}$
    • $D_{c/s} >= max{NF/u_s, F/d_{min}}$
    • 性质，随着N很大之后，主要看前一项$NF/u_s$, 线性增长
  • P2P模式
    • 服务器传输，最少一份，$F/u_s$
    • 接收端，$F/d_{min}$
    • 总体来看，客户端需要下载N份，总的需要上载量确定，则$NF/(u_s + \sum_{1}^{n}u_i)$
    • $D_{P2P} >= max{F/u_s, F/d_{min}, NF/(u_s + \sum_{1}^{n}u_i)}$
    • 性质，随着N增大，前两项不变，最后一项缓慢增大，上传量也在增加，因此不是线性增加

6.3 非结构化P2P

6.3.1 集中化目录 BitTorrent

  • tracker: 跟踪torrent中的参与节点
  • torrent：所有节点的集合
  • 一个块256K，bitmap指示有哪些文件块
  • 流程
    1. peer加入torrent，向tracker请求，tracker返回包含响应文件的主机的IP
    2. peer向这些IP请求文件快
    3. 过程中可能有人下线，也可能有人上线
  • 优先请求稀缺块
    • 前提，你给我的多，我也给你多一些
    • 当请求到稀缺块之后其他人会来请求，从而在他们那里下载得更多
  • tit-for tat
    • 优先疏通给我提供好的
    • 从等待的中找出给我提供带宽高的节点
  1. 问题
    1. 单点故障
    2. 性能瓶颈
    3. 侵犯版权

6.3.2 完全分布式 Gnutella

  • 过程
    • 建立起连接之后，向所有邻居发起查询
    • 邻居继续向其他发出查询
    • 拥有内容的节点发出响应
  • 限制泛洪
    • TTL
    • 记录已经经过响应的查询
  • 对等方加入
    • 加入后，向部分配置好的节点发送ping
    • 同样进行广播，收到的发出pong响应

7. CDN和视频流化服务

7.1 视频介绍

  • 视频由一序列图像组成 24 images/sec
  • 网络视频特点：高码率， 可以被压缩
  • 数字化图像，每个像素由若干bit标识
  • 编码：使用图像与图像之间的冗余来减低bit数
    • 空间冗余（图像内部）
    • 时间荣誉（相邻图像）
  • CBR：以固定速率编码
  • VBR：视频编码速率随时间变化

7.2 流化服务

  • DASH：dynamic adaptive streaming over HTTP
  • 服务器
    • 每个视频分割成多个块
    • 每个块独立储存，编码于不同码率
  • 客户端
    • 先获取告示文件
    • 周期性测试服务器到客户端的带宽
    • 查询告示文件，在一个时刻请求一个块
      • 带宽足够请求大码率
      • 不够时请求小码率
  • 客户端决定
    • 什么时候请求
    • 请求什么码率
    • 那里去请求

    再一个，罗翔老师谈到除了放弃对虚无和荒诞的抵抗，我们是不是还有一种幽默来对抗这种虚无和荒诞，这个立刻让我想到之前看的《瞬息全宇宙》还有《瑞士军刀男》，他们好像都是再通过另一种看似的荒诞来反抗我们让生活乃至人生的荒诞。瞬息全宇宙通过这一家人的拯救宇宙的看似有些随意得荒诞的故事阐述了选择以及爱等话题。瑞士军刀男通过那个已经死去的男人和这个活着的男人的莫名情感，对抗生活的毫无意义，军刀男拯救了他！！
